# Kindergarten Application — функциональность Version 1

Этот документ фиксирует фактически реализованное состояние проекта перед началом Version 2. Он описывает не планируемые возможности, а функции, существующие в кодовой базе на момент снимка.

## 1. Роли и доступ

В системе определены четыре роли:

- `SUPER_ADMIN`;
- `KINDERGARTEN_ADMIN`;
- `TEACHER`;
- `PARENT`.

Реализован следующий механизм доступа:

- после входа backend выдаёт JWT с `userId`, `tenantId` и списком ролей;
- JWT хранится во frontend в `localStorage` под ключом `token`;
- срок действия токена — 24 часа;
- frontend перенаправляет пользователя в кабинет согласно роли;
- role guards закрывают кабинеты от пользователей с неподходящей ролью;
- неавторизованный пользователь перенаправляется на `/login`;
- backend работает без HTTP-сессии и проверяет Bearer JWT на защищённых запросах;
- данные основных сущностей фильтруются по `tenantId` из контекста авторизованного пользователя.

Маршрутизация после входа:

| Роль | Начальная страница |
| --- | --- |
| `SUPER_ADMIN` | `/kindergarten-admin/dashboard` |
| `KINDERGARTEN_ADMIN` | `/kindergarten-admin/dashboard` |
| `TEACHER` | `/teacher/dashboard` |
| `PARENT` | `/parent/dashboard` |

Отдельного интерфейса для `SUPER_ADMIN` нет. Эта роль использует административный кабинет детского сада.

## 2. Публичная часть и аутентификация

### Главная страница

- краткое описание системы;
- кнопка перехода на вход;
- кнопка перехода на регистрацию.

### Регистрация

- регистрация по полному имени, email и паролю;
- проверка формы на frontend через Zod и React Hook Form;
- серверная проверка имени, email и длины пароля;
- нормализация email: удаление пробелов по краям и перевод в нижний регистр;
- проверка уникальности email;
- пароль хешируется BCrypt;
- публичная регистрация всегда создаёт пользователя с ролью `PARENT`;
- новый пользователь привязывается к tenant с ID `1`;
- после регистрации frontend автоматически выполняет вход и открывает кабинет родителя.

Ограничения полей регистрации:

| Поле | Правило |
| --- | --- |
| Полное имя | обязательно, от 2 до 50 символов |
| Email | обязательно, валидный email, уникальный во всей системе |
| Пароль | обязательно, от 6 до 50 символов |

### Вход и выход

- вход по email и паролю;
- получение JWT от `POST /auth/login`;
- автоматический переход в кабинет по роли;
- восстановление авторизации из `localStorage` после перезагрузки страницы;
- ручной выход с удалением JWT из браузера;
- endpoint `/auth/me` возвращает ID и роли текущего пользователя.

Не реализованы refresh token, восстановление пароля, подтверждение email, MFA и управление активными сессиями.

## 3. Кабинет администратора детского сада

Доступен для `KINDERGARTEN_ADMIN` и `SUPER_ADMIN`.

### Dashboard администратора

- количество детей;
- количество групп;
- количество воспитателей;
- общая статистика посещаемости по статусам `PRESENT`, `ABSENT`, `SICK`;
- вычисление процента присутствующих;
- круговая диаграмма посещаемости;
- быстрые переходы к управлению воспитателями и группами;
- состояния загрузки и ошибки получения статистики.

### Управление воспитателями

- постраничный список воспитателей, по 10 записей в интерфейсе;
- отображение полного имени, email и назначенной группы;
- создание воспитателя;
- редактирование имени и email;
- необязательная смена пароля при редактировании;
- удаление с подтверждением;
- уведомления об успешных и неуспешных операциях;
- отображение пустого состояния и ошибок загрузки.

Бизнес-правила:

- email нормализуется и должен быть уникальным;
- имя нормализуется: лишние пробелы удаляются;
- создаваемый пользователь автоматически получает роль `TEACHER`;
- удалить воспитателя, назначенного активной группе, нельзя;
- при редактировании пустой пароль сохраняет текущий пароль.

### Управление группами

- постраничный список групп, по 10 записей в интерфейсе;
- создание группы;
- редактирование группы;
- удаление группы с подтверждением;
- поля группы: название, возрастной диапазон, воспитатель;
- воспитателя можно не назначать;
- список выбора показывает только доступных воспитателей;
- при редактировании текущий воспитатель остаётся доступным для выбора;
- просмотр связанной информации в таблице;
- состояния загрузки, пустого результата и ошибок.

Бизнес-правила:

- название обязательно, максимум 255 символов;
- возрастной диапазон необязателен, максимум 255 символов;
- пробелы в названии и возрастном диапазоне нормализуются;
- один активный воспитатель может быть назначен только одной активной группе;
- назначить пользователя без роли `TEACHER` или из другого tenant нельзя;
- удаление группы выполнено как soft delete;
- при удалении воспитатель отсоединяется от группы;
- дети удалённой группы не удаляются, а становятся не назначенными ни в одну группу.

### Управление детьми

- отдельный список детей без группы;
- постраничный список всех детей, по 10 записей;
- отображение имени, фамилии, даты рождения и группы;
- просмотр связанных родителей с именем и email;
- просмотр подробностей группы ребёнка;
- назначение ребёнка в группу;
- перенос ребёнка между группами;
- снятие назначения группы;
- обновление обоих списков после изменения;
- уведомления об успешных и неуспешных операциях.

Создание, редактирование и удаление ребёнка из административного UI отсутствуют, хотя соответствующие backend endpoints существуют.

## 4. Кабинет воспитателя

Доступен только для `TEACHER`.

Для функций класса и журнала воспитатель должен быть назначен группе. Если назначенной группы нет, backend возвращает ошибку или пустое/ошибочное состояние в UI.

### Dashboard воспитателя

- приветственный рабочий экран воспитателя;
- информация о группе и детях;
- отображение последних записей дневника;
- быстрые действия для работы с дневником;
- локальные личные заметки воспитателя;
- добавление, редактирование и удаление локальных заметок;
- заметки сохраняются только в `localStorage` браузера под ключом `teacher_notes` и не синхронизируются с backend.

### Class Records

- список детей только из назначенной воспитателю группы;
- таблица с именем, фамилией, датой рождения и названием группы;
- состояния загрузки, ошибки и пустой группы;
- данные родителей входят в backend response, но на этом экране не показываются.

### Посещаемость

- выбор ребёнка из назначенной группы;
- переключение между месяцами;
- календарь выбранного месяца;
- цветовое отображение статусов по дням;
- просмотр записей за выбранный месяц;
- создание записи посещаемости;
- изменение существующей записи за дату;
- удаление/сброс записи;
- доступные статусы: `PRESENT`, `ABSENT`, `SICK`;
- отображение месячной статистики и последних записей;
- автоматическое обновление UI после сохранения или удаления.

### Ежедневный журнал

- создание записи для группы воспитателя;
- дата записи, по умолчанию текущая;
- текстовое резюме дня (`summary`);
- достижения и важные события (`milestones`);
- загрузка нескольких фотографий и сохранение их URL;
- список всех записей группы в обратном порядке по дате;
- просмотр отдельной записи;
- редактирование даты, резюме, достижений и списка фотографий;
- доступ воспитателя только к собственным записям;
- состояния загрузки и ошибок в интерфейсе.

Удаление записи журнала не реализовано.

### Загрузка фотографий

- загрузка доступна только воспитателю;
- поддерживаются расширения `.png`, `.jpg`, `.jpeg`, `.webp`;
- MIME type должен начинаться с `image/`;
- максимальный размер одного файла — 5 MB;
- имя файла очищается и дополняется UUID;
- файлы сохраняются в локальную директорию backend;
- загруженные изображения публично доступны по URL `/uploads/{filename}`.

## 5. Кабинет родителя

Доступен только для `PARENT`.

### Dashboard родителя

- персональное приветствие с именем пользователя;
- список связанных детей;
- возраст и группа ребёнка;
- переход к профилю выбранного ребёнка;
- добавление нового ребёнка;
- лента последних пяти доступных записей дневника;
- переход к полной ленте и отдельной записи;
- состояния загрузки, ошибок и пустых данных.

### Дети родителя

- родитель видит только детей, связанных с его user ID;
- выбор ребёнка, если их несколько;
- профиль с именем, фамилией, датой рождения, рассчитанным возрастом и группой;
- создание ребёнка с автоматической связью с текущим родителем;
- редактирование имени, фамилии и даты рождения;
- вкладка посещаемости;
- переход обратно на dashboard.

Правила данных ребёнка:

- имя и фамилия обязательны, максимум 100 символов каждое;
- дата рождения должна быть в прошлом;
- group ID, если задан, должен быть положительным;
- родитель не может читать или изменять чужого ребёнка;
- удаление ребёнка поддерживается backend как soft delete, но кнопки удаления в кабинете родителя нет.

### Посещаемость ребёнка

- календарь посещаемости по месяцам;
- переход между месяцами;
- просмотр записей `PRESENT`, `ABSENT`, `SICK`;
- родитель может отметить `ABSENT` или `SICK`;
- родитель может изменять и удалять собственно доступные записи;
- запись за одну дату обновляется вместо создания дубликата;
- если удалённая запись за эту дату существует, повторное сохранение восстанавливает её;
- родитель не может устанавливать статус `PRESENT`;
- родитель не может создавать или менять запись за прошедшую дату;
- родитель не может редактировать или удалить запись со статусом `PRESENT`;
- стандартный диапазон запроса — текущий месяц;
- максимальный диапазон одного запроса — 92 дня.

### Дневник

- лента записей только для групп, в которых состоят дети родителя;
- просмотр даты, резюме, достижений и фотографий;
- просмотр отдельной записи;
- проверка доступа к записи через связь parent → child → group;
- родитель не может создавать или редактировать записи дневника.

### Не завершённые вкладки родителя

- вкладка `Development` отображает только placeholder; отдельной модели развития ребёнка нет;
- пункт `Messages` присутствует в навигации, но страницы `/parent/messages` и backend-модуля сообщений нет.

## 6. Общая логика посещаемости

- одна логическая запись посещаемости на ребёнка и дату;
- повторный `POST` на ту же дату обновляет найденную запись;
- статусы: присутствовал, отсутствовал, болел;
- все операции ограничены tenant;
- администратор может получить постраничный список всех записей tenant;
- остальные роли получают записи только через конкретного доступного ребёнка;
- воспитатель имеет доступ только к детям своей группы;
- родитель имеет доступ только к связанным детям;
- удаление является soft delete через `deletedAt`;
- сводная статистика считает записи по каждому статусу для tenant.

## 7. Общая логика детей и связей с родителями

- ребёнок принадлежит tenant;
- ребёнок может не иметь группы;
- ребёнок может иметь нескольких родителей через связующую таблицу;
- response ребёнка содержит краткие данные всех связанных родителей;
- ребёнок, созданный родителем, автоматически связывается с этим родителем;
- администратор видит всех детей tenant;
- родитель видит только связанных детей;
- воспитатель видит только детей своей группы;
- попытка обратиться к недоступному ребёнку маскируется как `404 Child not found`;
- удаление ребёнка является soft delete.

В сервисе существует внутренняя операция добавления связи родителя с ребёнком, но отдельный публичный controller endpoint и UI для управления такими связями отсутствуют.

## 8. Tenant-функциональность

Backend содержит CRUD API для tenants:

- получить список;
- получить tenant по ID;
- создать tenant;
- изменить tenant;
- выполнить soft delete tenant.

Модель tenant содержит название, адрес, контактную информацию и тарифный план.

Ограничения V1:

- UI управления tenants отсутствует;
- onboarding нового детского сада отсутствует;
- публичная регистрация жёстко использует tenant ID `1`;
- миграция создаёт один `Default Kindergarten` для MVP;
- отдельный полноценный workflow `SUPER_ADMIN` не реализован.

## 10. Общие технические функции

### Backend

- REST API на Spring MVC;
- Spring Security и собственный JWT filter;
- BCrypt для паролей;
- JPA/Hibernate repositories;
- DTO и MapStruct mapping;
- Bean Validation;
- единый обработчик основных ошибок с ответами `400`, `403`, `404`, `409`;
- пагинация Spring Data;
- soft delete для детей, групп, посещаемости и tenants;
- Flyway migrations при запуске;
- Swagger UI и OpenAPI JSON;
- CORS для локального frontend и production-домена;
- tenant context из JWT;
- логирование операций с детьми;
- Actuator dependency подключена.

### Frontend

- Next.js App Router;
- TypeScript;
- отдельные layouts и navigation для каждой роли;
- Material UI и адаптивные layouts;
- React Hook Form + Zod для форм;
- shared UI-компоненты: buttons, inputs, dialogs, confirmations, tables, pagination, loaders, empty/error states, notifications;
- состояния загрузки, ошибок и пустых выборок на основных экранах;

## 11. Что не считается реализованной функциональностью V1

## 12. Ключевые ориентиры для сравнения с Version 2

При разработке V2 изменения удобно фиксировать относительно следующих границ V1:

1. один технически предустановленный tenant;
2. четыре роли, но только три полноценных пользовательских кабинета;
3. JWT без refresh token;
4. локальное хранение фотографий;
5. посещаемость, группы, дети, воспитатели и групповой журнал как основные домены;
6. сообщения и индивидуальное развитие ребёнка не реализованы;
7. часть backend CRUD возможностей не представлена в UI;
8. CI/CD ориентирован на один production-сервер через Docker Compose и SSH.
